# Gap: STORY-001 — 2026-06-14
## What changed
The `GET /api/profile/sections` endpoint was implemented per the story spec, reading data from `file.data.gpa`, `file.data.school`, etc. (spec-defined nested structure written by manual entry endpoints). However, STORY-003 (document extraction) was implemented with `profile-merge.js`, which writes extracted data at the **top level** of each profile JSON file (e.g., `file.gpa`, `file.schoolName`, `file.courses`, `file.achievements[]`, `file.activities[]`) rather than inside the `data:` envelope. As a result, `readSectionAcademic()`, `readSectionTests()`, and `readSectionItems()` in `profile.js` all failed to detect real data as non-empty. The fix adds top-level field detection to all three reader functions and builds human-readable summary strings that work with both the manual-entry schema and the extraction schema.

## Files affected
- `/Users/mangeshpise/Code/AO/src/server/routes/profile.js` — Fixed `readSectionAcademic()`, `readSectionTests()`, `readSectionItems()` to handle both the spec data structure (nested `file.data.*`) and the extraction data structure (top-level `file.*` fields written by `profile-merge.js`). Added `buildAcademicSummary()` and `buildTestsSummary()` helpers.
- `/Users/mangeshpise/Code/AO/src/public/js/app.js` — Applied `escapeHtml(String(statusLabel))` to the section card status label to prevent `[object Object]` display when the backend returns a summary object instead of a string.

## Side-effects on other stories
- STORY-003: The `profile-merge.js` top-level write schema is now explicitly accommodated in the sections endpoint. Any future story that reads academic/tests/achievements/activities data must be aware of both schemas coexisting in the same JSON file.
- STORY-005 (export): The export story should use `readProfileSection()` from `profile-merge.js` (which reads top-level fields) rather than reading the `data:` envelope to get extracted data.

## Recommended spec update
`story-spec.md` for STORY-001 (and STORY-003) should clarify that `profile-merge.js` writes extraction data to the **top level** of section JSON files, while manual entry endpoints write into the `data:` envelope. The `GET /api/profile/sections` implementation must handle both schemas. Alternatively, the architecture should standardize on one schema — preferably updating `profile-merge.js` to write into `file.data.*` to match the spec-defined envelope.
