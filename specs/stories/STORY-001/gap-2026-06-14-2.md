## What changed
The story spec (v1.1.2) defines AP/IB exam scores as stored in `tests.json` under `data.ap[]` and `data.ib[]`, each item having an `id` field for per-item CRUD. However, the existing manual entry endpoint `POST /api/profile/academic/add-exam-score` stores scores in `academic.json` under a top-level `apIbScores[]` array. This means the new CRUD endpoints (`GET /api/profile/academic/exam-scores`, `PUT /api/profile/academic/exam-score/:id`, `DELETE /api/profile/academic/exam-score/:id`) must search both `academic.json.apIbScores[]` and `tests.json.data.ap[]`/`tests.json.data.ib[]` to find a score by ID. The `GET /api/profile/academic/exam-scores` endpoint merges both arrays into unified `ap` and `ib` lists in its response. The `PUT` and `DELETE` endpoints try `academic.json` first, then fall back to `tests.json`. The Academic detail page UI (`showAcademicDetail`, `_renderExamScoresSection`) now uses `GET /api/profile/academic/exam-scores` to get the complete list with IDs for Edit/Delete buttons, rather than relying on the section data from `GET /api/profile/section/academic` which may not expose all IDs.

## Recommended spec update
The story spec should clarify that AP/IB exam scores are stored in `academic.json.apIbScores[]` (as written by the current `POST /api/profile/academic/add-exam-score` endpoint), not in `tests.json` as v1.1.2 specifies. Alternatively, migrate the `add-exam-score` endpoint to write to `tests.json.data.ap[]`/`tests.json.data.ib[]` to match the spec. Until migration, all code that reads AP/IB scores must check both locations.
